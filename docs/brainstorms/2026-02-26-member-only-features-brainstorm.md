## <span data-proof="authored" data-by="ai:claude">What We're Building</span>

<span data-proof="authored" data-by="ai:claude">A member-only feature layer for the Sovereign Hope app that provides:</span>

2. **<span data-proof="authored" data-by="ai:claude">Daily Prayer Partners</span>** <span data-proof="authored" data-by="ai:claude">- Each member sees 3 randomly-assigned members to pray for daily, unique per member, regenerated each day</span>

## <span data-proof="authored" data-by="ai:claude">Why This Approach</span>

### <span data-proof="authored" data-by="ai:claude">Member Management</span>

- **<span data-proof="authored" data-by="ai:claude">Staff manages membership via Firebase Console</span>** <span data-proof="authored" data-by="ai:claude">- No in-app admin UI initially. Staff sets custom claims and creates member profiles directly. This avoids building admin infrastructure before proving the features are valuable.</span>

- **<span data-proof="authored" data-by="ai:claude">Firebase custom claims for the member flag</span>** <span data-proof="authored" data-by="ai:claude">- Custom claims are embedded in the auth token, verified server-side in Firestore rules via</span> <span data-proof="authored" data-by="ai:claude">`request.auth.token.isMember == true`. They cannot be spoofed or self-assigned by clients. This is the gold standard for role-based access in Firebase.</span>

### <span data-proof="authored" data-by="ai:claude">Data Architecture (Approach A: Flat Collections)</span>

- **<span data-proof="authored" data-by="ai:claude">`members/{uid}`</span>** <span data-proof="authored" data-by="ai:claude">- Profile data (name, photo) managed by staff when granting membership</span>

- **<span data-proof="authored" data-by="ai:claude">`prayerAssignments/{date}/{uid}`</span>** <span data-proof="authored" data-by="ai:claude">- Daily prayer assignments generated by Cloud Function</span>

- <span data-proof="authored" data-by="ai:claude">Separate top-level collections for clean separation of concerns, simple queries, and straightforward security rules</span>

### <span data-proof="authored" data-by="ai:claude">Security Model (Two Layers)</span>

1. **<span data-proof="authored" data-by="ai:claude">Client-side navigation guards</span>** <span data-proof="authored" data-by="ai:claude">- Check</span> <span data-proof="authored" data-by="ai:claude">`isMember`</span> <span data-proof="authored" data-by="ai:claude">custom claim before rendering member screens. Prevents UI access.</span>
2. **<span data-proof="authored" data-by="ai:claude">Firestore security rules</span>** <span data-proof="authored" data-by="ai:claude">-</span> <span data-proof="authored" data-by="ai:claude">`members`</span> <span data-proof="authored" data-by="ai:claude">and</span> <span data-proof="authored" data-by="ai:claude">`prayerAssignments`</span> <span data-proof="authored" data-by="ai:claude">collections require</span> <span data-proof="authored" data-by="ai:claude">`request.auth.token.isMember == true`. Hard data-level enforcement even if client is bypassed.</span>

### <span data-proof="authored" data-by="ai:claude">Prayer Assignment Generation</span>

- **<span data-proof="authored" data-by="ai:claude">Scheduled Cloud Function (midnight daily)</span>** <span data-proof="authored" data-by="ai:claude">- Pre-generates 3 random prayer partners for every active member. Assignments are ready before anyone opens the app.</span>

- <span data-proof="authored" data-by="ai:claude">Each member gets 3 unique, random other members (not themselves)</span>

- <span data-proof="authored" data-by="ai:claude">Assignments vary per member (not the same 3 for everyone)</span>

### <span data-proof="authored" data-by="ai:claude">Privacy & Opt-out</span>

- **<span data-proof="authored" data-by="ai:claude">No opt-out</span>** <span data-proof="authored" data-by="ai:claude">- All members appear in the directory and prayer rotation. Membership implies participation.</span>

### <span data-proof="authored" data-by="ai:claude">Profile Data Source</span>

- **<span data-proof="authored" data-by="ai:claude">Dedicated member profiles in Firestore</span>** <span data-proof="authored" data-by="ai:claude">- Staff enters name and photo when granting membership. Decoupled from auth provider data (which can be inconsistent or generic).</span>

- <span data-proof="authored" data-by="ai:claude">Directory shows name and photo only initially. May expand later.</span>

## <span data-proof="authored" data-by="ai:claude">Proposed Firestore Schema</span>

```
members/{uid}
  ├── displayName: string
  ├── photoURL: string | null
  ├── createdAt: Timestamp
  └── uid: string

prayerAssignments/{date}/{uid}
  ├── memberIds: [uid1, uid2, uid3]
  └── generatedAt: Timestamp
```

## <span data-proof="authored" data-by="ai:claude">Proposed Firestore Rules (additions)</span>

```
match /members/{memberId} {
  allow read: if request.auth != null && request.auth.token.isMember == true;
  allow write: if false;  // managed by staff/Admin SDK only
}

match /prayerAssignments/{date}/{memberId} {
  allow read: if request.auth != null
               && request.auth.token.isMember == true
               && request.auth.uid == memberId;
  allow write: if false;  // managed by Cloud Function only
}
```

## Resolved Questions

1. **Timezone for scheduled function** - Midnight Mountain Time (America/Boise), the church's local timezone.
2. **Token refresh UX** - Force refresh auth token on app foreground. Automatically picks up claim changes within one app session cycle.
3. **Photo storage** - Firebase Storage. Upload photos, store download URL in the member doc. Stays within Firebase ecosystem.
4. **Member removal** - Immediate loss of access on next token refresh (foreground). No grace period. Clean cutoff.
5. **Prayer assignment edge cases** - Assign as many as possible. If fewer than 4 total members, assign 1-2 instead of 3. UI adapts to show however many are available.
